Phase 0 — Comparative Benchmarking walkthrough

andmeter · generated from the working tree (uncommitted) · 2026-06-23 JST

Turns &Meter from a single-target load-tester into a cross-provider comparative platform. Phase 0 lands the minimum scaffolding plus one real slice: a performance head-to-head on GLM 5.2 — myorg staging glm-5.2 vs Together AI zai-org/GLM-5.2. Read top-to-bottom: the order follows the data flow, not the file tree.

Decisions baked in

  • Extended the dormant benchmark_endpoints/benchmark_models registry rather than building new tables.
  • api_key is stored plaintext in MySQL (matches the conformance/agentic precedent, staging-internal) but never returned by the API — only has_api_key + a masked suffix. Benchmark runners read keys from env, not from the catalog.
  • Cost is computed inline at analysis time (a precursor to the Phase 1 cost-joiner). A "run-set/suite" abstraction that fans one config across N providers is deferred to Phase 1 — Phase 0 pairs tasks manually.
  • Reused the existing ResultComparison page; the only gaps were provider labeling and cost.

Schema — new catalog columns

migration 011 · init_db.sql

What

Adds provider auth (api_key, provider) to benchmark_endpoints and per-model pricing to benchmark_models.

Why

The registry existed but carried no auth and no pricing — both are prerequisites for targeting competitors and joining cost. Pricing lives on the model (the correct grain); the key on the endpoint.

backend/migrations/011_endpoint_auth_pricing.sql
ALTER TABLE `benchmark_endpoints` + ADD COLUMN `api_key` varchar(512) DEFAULT NULL + COMMENT 'Provider Bearer token (plaintext; staging-internal; masked on read)' AFTER `description`, + ADD COLUMN `provider` varchar(64) DEFAULT NULL AFTER `api_key`; ALTER TABLE `benchmark_models` + ADD COLUMN `input_price_per_1m` double DEFAULT NULL, + ADD COLUMN `output_price_per_1m` double DEFAULT NULL, + ADD COLUMN `cached_input_price_per_1m` double DEFAULT NULL, + ADD COLUMN `currency` varchar(8) NOT NULL DEFAULT 'USD';
Verify: migration ↔ init_db.sql columns are identical (name/type/collation/default). MySQL 5.7 has no ADD COLUMN IF NOT EXISTS, so migrations are a one-off manual apply; init_db.sql mirrors the DDL for fresh DBs.

ORM + Pydantic — typed contract

backend/model/endpoint.py

What

Mirrors the columns on the SQLAlchemy models and the request/response schemas. The response exposes has_api_key + api_key_suffix — never the raw key.

Why

Masking the secret at the schema boundary makes it structurally impossible to leak it through any read path.

backend/model/endpoint.py — ORM
description = Column(Text, nullable=True) + api_key = Column(String(512), nullable=True) + provider = Column(String(64), nullable=True) ... + input_price_per_1m = Column(Float, nullable=True) + output_price_per_1m = Column(Float, nullable=True) + cached_input_price_per_1m = Column(Float, nullable=True) + currency = Column(String(8), nullable=False, server_default="USD")
backend/model/endpoint.py — EndpointResponse (no raw key)
description: Optional[str] = None + has_api_key: bool = False + api_key_suffix: Optional[str] = None + provider: Optional[str] = None
Verify: no response model carries a raw api_key field — confirmed across EndpointResponse/ModelResponse.

Service — persist + mask the key

backend/service/endpoint_service.py

What

Persists the new fields, masks the key on read, and on update overwrites the key only when one is actually supplied.

Why

An operator can edit an endpoint without re-typing the secret (blank = keep existing), and the raw key never crosses the HTTP boundary.

backend/service/endpoint_service.py — _build_endpoint_response
+ api_key = str(endpoint.api_key) if endpoint.api_key is not None else None + api_key_suffix = "…" + api_key[-4:] if api_key else None return EndpointResponse( + has_api_key=bool(endpoint.api_key), + api_key_suffix=api_key_suffix, + provider=str(endpoint.provider) if endpoint.provider is not None else None, ...)
backend/service/endpoint_service.py — update guard
setattr(endpoint, "provider", payload.provider) + # A blank api_key on update preserves the existing key. + if payload.api_key: + setattr(endpoint, "api_key", payload.api_key)
Verify: api_key[-4:] is slice-safe on short keys; currency defaults to USD; the raw key is never returned on any path (create passes an empty model list, update re-reads).

Cost join — pricing × measured tokens

backend/service/analysis_service.py

What

Resolves provider + per-1M pricing from the catalog by matching URL host and model, then estimates a per-request USD cost from the measured token counts.

Why

A thin, inline precursor to the Phase 1 cost-joiner — enough to make the head-to-head meaningful without a new module.

backend/service/analysis_service.py — _resolve_catalog_pricing
target_hostname = urlparse(target_host).hostname for endpoint in endpoint_result.scalars().all(): + if urlparse(endpoint.base_url).hostname == target_hostname: + matched_endpoint = endpoint # base_url often ends in /v1; host-only match + break
backend/service/analysis_service.py — cost
+ input_tokens = max(total_tokens - completion_tokens, 0) + cost_per_request_usd = ( + input_tokens * input_price_per_1m + + completion_tokens * output_price_per_1m + ) / 1_000_000
Verify: priced runs populate the field; a missing price or token count yields None (guarded, no crash). The urlparse(...).hostname match is case-insensitive (Python lowercases the host) — verified empirically.

Comparison contract — carry the new fields

llm_task.py · llm_task_service.py

What

Adds three optional fields to ComparisonMetrics and threads them through the explicit constructor that builds the API response.

Why

The constructor is field-by-field (not **dict), so the new fields must be passed by hand — using .get() keeps it safe when keys are absent.
