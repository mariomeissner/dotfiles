Prefix caching vs. PD disaggregation

Two different optimizations that get mixed up because they both live near "the KV cache" — one reuses work, the other splits hardware. They're orthogonal, and only one runs in ai&'s prod today.

Why this matters — ai&'s stack has two separate prefix caches (engine-level and control-plane) plus a dormant PD-disaggregation path. Conflating any of these leads to wrong assumptions about what's actually reused, what's actually split, and what's actually running in prod.
Two chains hang off the same root: caching (left, reuses work) and disaggregation (right, splits hardware — dashed because it's not live).

1. Prefill vs. decode

foundational

Serving one LLM request has two phases with very different compute profiles. Prefill processes your entire input prompt in one parallel forward pass — it's compute-bound (lots of matmuls, GPU-bound). Decode then generates the output one token at a time, each step needing the whole model's attention state again — it's memory-bandwidth-bound, not compute-bound, because each step does little new math but has to read a lot of state.

In this stack: every ai& model today runs both phases on the same vLLM replica — one InferenceService does prefill and decode back-to-back for a request. Dashboard panels labeled role=prefill / role=decode are just per-phase metrics on that one pod, not two separate pools.

→ Decode's need to "read state again" every step is exactly what the KV cache exists to make cheap.

2. KV cache

foundational

Instead of recomputing attention over the whole sequence at every decode step, the engine stores each token's key/value projections once (in GPU memory) and reuses them. Decode then only computes attention for the *new* token against that stored history. This is what makes decode tractable — without it, generating token N would cost as much as prefilling N tokens.

Why it matters for caching: if two different requests share the same prompt prefix (e.g. the same long system prompt), their KV state for that prefix is identical — computing it twice is wasted work. That observation is the seed of both caching layers below.

→ vLLM turns that observation into an actual cache, shared across requests, not just across steps of one request.

3. vLLM engine-level prefix caching

derived

vLLM (the engine ai& runs) keeps KV cache blocks around after a request finishes and reuses them for the next request if its prompt shares a prefix — so a repeated system prompt or few-shot preamble doesn't get recomputed. This lives entirely inside the GPU/engine layer, keyed in fixed-size blocks (vLLM's default block size is 16 tokens).

In this stack: Moreh's Heimdall gateway does prefix-cache-aware routing — it tries to send a request to whichever replica already has that prompt's prefix warm in its KV cache, across the pool of otherwise-homogeneous replicas.
Don't confuse this with #4 below — this is real GPU memory being reused; the next layer up is a completely separate cache that never touches GPU state.

→ uni adds its own, unrelated cache on top, for a different purpose: billing, not GPU memory.

4. uni's control-plane prompt cache (billing)

derived

Separately from anything vLLM does, uni's gateway (worker-inference src/lib/prompt-cache.ts) keeps a per-org Redis table of chained SHA-256 hashes of prompt prefixes. When a new request's prefix matches a prior one, it looks up that prior request's real prompt_tokens count for the matching length and charges those tokens at the model's discounted cachedInputPer1m rate instead of full input price.

Mechanics: TTL 600s, minimum 1024 cacheable tokens, 128-token block granularity (note: different from vLLM's own 16-token block size above — these are two independent systems that happen to both use "blocks"). Applies to both streaming and non-streaming requests.
Gotcha: streaming responses don't surface cached_tokens in the standard usage object (only the non-stream path gets it, or an opt-in event: metrics SSE most clients don't request) — so cost tools reading response usage will show cached=0 even when the discount was actually applied. Real cost lives in uni's GET /logs, not client-side usage.

→ That's the whole "caching" branch. The other branch off prefill/decode is about hardware, not reuse — PD disaggregation.

5. PD disaggregation

derived

Because prefill is compute-bound and decode is memory-bandwidth-bound, they want different hardware ratios. PD disaggregation splits them onto separate GPU pools — a small pool of prefill-only replicas feeding a larger pool of decode-only replicas, with KV state transferred between pools over RDMA (e.g. NIXL) — so each phase can be sized and scaled independently instead of being stuck on one monolithic replica shape.

Every live model today: one replica shape does both phases. Heimdall only does prefix-cache routing across these homogeneous replicas.
Staged but dormant: platform's prod manifests already hold ~200 prefill/decode InferenceServiceTemplate defs with NIXL config — but they target AMD MI300/MI308X, not ai&'s NVIDIA fleet, and nothing binds them. Real PD disaggregation only runs today in InferenceX benchmark recipes. Bringing it to prod is mostly Moreh's work (NVIDIA templates) plus platform (split pools, RDMA, per-pool KEDA) — worthwhile mainly for large MoE models at high concurrency; deferred to benchmarking Milestone 5.

→ Back to the original question: caching and disaggregation are independent axes — ai& runs prefix caching (both layers) on a non-disaggregated fleet, which is a perfectly valid combination.
